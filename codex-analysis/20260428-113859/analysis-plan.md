## Stack Summary
- Project type: single-package Python HTTP server named `exphttp`, version `2.0.0`, source under `src/`, package entry point `exphttp = src.cli:main`.
- Runtime: Python `>=3.10` with standard-library core; optional `cryptography>=44.0` extra enables AES-GCM and ECDH Secure Notepad features.
- Protocol surface: raw socket HTTP/1.1 server, custom methods (`FETCH`, `INFO`, `PING`, `NONE`, `NOTE`, `SMUGGLE`), upload methods (`POST`, `PUT`, `PATCH`, `NONE`), opt-in unknown-method advanced upload, WebSocket support for `/notes/ws`.
- Storage model: filesystem-backed `uploads/` for user files and `notes/` for encrypted note blobs and metadata. No database, cache, queue, ORM, or external runtime service was detected.
- Security features: uploads-only path containment with `Path.resolve().relative_to()`, Basic Auth with PBKDF2-SHA256, IP-based auth rate limiting, TLS 1.2+ via `ssl.SSLContext`, optional Let's Encrypt via `certbot`, CSP/nosniff headers, WebSocket origin checks.
- Frontend: bundled static HTML/CSS/JavaScript under `src/data/`; no npm package manifest. Browser smoke uses Playwright through `npx` or a bundled Codex Playwright CLI.
- Build/package: setuptools backend, dynamic version from `src.__version__`, typed package marker `src/py.typed`, package data for bundled UI assets.
- Tests/quality: pytest 9 configuration in `pyproject.toml`, Hypothesis property tests, pytest-cov coverage gate in CI, strict mypy, Ruff lint/format, pip-audit and Bandit security workflow.
- Docs: MkDocs + Material with mirrored root docs generated by `tools/sync_docs.py`; ADRs and threat model are present.
- Infra/deploy: multi-stage Dockerfile pinned to a Python base image digest; Docker runs as non-root and includes HTTP healthcheck; GitHub Actions CI/security/dependabot/pre-commit are present.
- Package manager signal: `constraints/ci.txt` is tracked and used by CI/Docker; untracked `uv.lock` exists and may introduce local package-resolution ambiguity if adopted without policy.

## Observed Problems & Risks
- [MEDIUM] WebSocket parser accepts unmasked client frames even though client-to-server masking is required by RFC 6455. Evidence: `src/websocket.py:81` says masking is required, but `parse_ws_frame()` accepts `masked == False` and returns `raw_payload` at `src/websocket.py:121-130`. Tests should verify rejection of unmasked client frames and unsupported/opcode edge cases.
- [MEDIUM] Gzip post-processing can defeat streaming by reading an entire streamed file into memory. Evidence: `src/server.py:262-274` calls `response.stream_path.read_bytes()` for compressible streamed responses before converting them into buffered bodies; this conflicts with the streaming behavior described in `CHANGELOG.md:51-56` and can create avoidable memory pressure for large text/JSON/SVG files.
- [MEDIUM] CORS handling appears inconsistent for multiple allowed origins. Evidence: WebSocket origin checks split `self.cors_origin` on commas in `src/server.py:559-563`, but HTTP response finalization writes the raw `cors_origin` string directly to `Access-Control-Allow-Origin` in `src/http/response.py:108-109`, which will be invalid if the CLI receives a comma-separated list.
- [MEDIUM] Documentation and AI guidance are stale in ways that can mislead contributors into reintroducing removed insecure patterns. Evidence: `CLAUDE.md:22` documents removed `--opsec --sandbox` flags; `CLAUDE.md:48`, `CLAUDE.md:61-64`, and `CLAUDE.md:115` describe `OpsecHandlersMixin`/`OPSEC_*` handlers no longer present; `CLAUDE.md:95` recommends the path-prefix `startswith()` check explicitly rejected by ADR-004. `CHANGELOG.md:35` says advanced upload is enabled by default, but `src/cli.py:73-78` defaults `--advanced-upload` to false.
- [MEDIUM] Tooling versions drift between pre-commit and CI constraints. Evidence: `.pre-commit-config.yaml:14-22` pins Ruff `v0.9.0` and mypy `v1.14.0`, while `constraints/ci.txt:64` pins Ruff `0.15.5` and the same file pins mypy `1.20.1`; local hooks may disagree with CI.
- [LOW] Error metrics may under-report application errors. Evidence: `RequestPipeline.process()` records dispatched 4xx/5xx responses with `error=False` at `src/request_pipeline.py:120-121`, records direct auth/size responses without an error flag at `src/request_pipeline.py:104-115`, and only increments `total_errors` for exceptions at `src/request_pipeline.py:139-145`. If `total_errors` is intended as operational error rate, this needs a clearer contract.
- [LOW] Package-manager policy is ambiguous. Evidence: tracked `constraints/ci.txt` is the documented reproducibility mechanism, while untracked `uv.lock` exists and contains newer dependency pins (for example `cryptography 47.0.0`, `mypy 1.20.2`) than `constraints/ci.txt`; this should be intentional before it lands.
- [LOW] Some naming remains historically `opsec` in frontend/test tooling while user-facing docs use advanced upload. Evidence: `tools/browser_smoke.py` fixture names and UI IDs/classes still use `opsec`; this may be acceptable as internal compatibility, but documentation agents should decide whether it is intentional legacy naming or confusing debt.
